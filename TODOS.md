## TODO: Parallel API calls with concurrency cap

**What:** Implement a concurrency limiter (max 3 parallel Claude calls) in the background fill loop using `p-limit` or a simple semaphore.

**Why:** Sequential fills on 25-field forms (YC) take 75-150 seconds. Parallel calls at concurrency 3 cut that to 25-50 seconds.

**Pros:** 3x faster fills on large forms. Dramatically better UX on YC.

**Cons:** More complex error handling (individual field failures in a concurrent batch). Badge counter becomes harder to display correctly during parallel fills.

**Context:** Sequential is intentional in v1 for simplicity and rate-limit safety. After Weekend 3 validates the sequential baseline on real forms, evaluate whether the 90-second wait is actually a problem in practice before building concurrency. Don't optimize prematurely.

**Depends on / blocked by:** v1 working baseline (Weekend 3 validation).

## TODO: Resume from interruption (progress persistence)

**What:** Persist `{ status: 'in-progress' | 'done', current: N, total: M, failedSelectors: string[] }` to `browser.storage.local` during fill. On popup open, read this state and show "Fill was interrupted at field 10 — resume?" if in-progress.

**Why:** Safari's background service worker has an aggressive kill timer. If killed mid-fill, the user currently has no indication that the fill stopped. Skip-if-filled handles the re-run case, but the user has to guess why the form is half-filled.

**Pros:** Clear recovery path. User knows the fill stopped and exactly where.

**Cons:** ~30 extra lines. Adds storage read on popup open. Storage cleanup needed after a successful fill.

**Context:** Only worth building if the kill-timer actually fires in Weekend 3 testing. Each Claude API call resets the idle timer, so the risk may be lower than the worst-case 30-45s window. Don't build this speculatively — validate first in Weekend 3.

**Depends on / blocked by:** Weekend 3 testing on real forms with >15 fields.

## TODO: Chat refinement loop (v2 primary target)

**What:** In-popup chat per field: after filling, user sees each field's answer in the popup. Typing "make this more concise" in a per-field chat input triggers a Claude rewrite with the original profile + original answer + refinement instruction as context. Writes the revised answer back to the form.

**Why:** The gap between a good AI-written answer and a great personalized answer is exactly what chat refinement closes. This was the original core differentiator, descoped from v1 to reduce complexity and API costs.

**Pros:** Core differentiator. Nobody in this space has built this for Safari. The popup architecture already has a results view after fill — adding a chat input per field is the natural v2 extension.

**Cons:** Requires a more complex popup UI (field results list + per-field chat input). Adds API calls per refinement (1-3 calls per refined field per session).

**Context:** Build the fill loop first (v1). Validate that the core fill quality is good enough that refinement is worth having. If v1 answers are 90% right already, refinement may not be needed. If they're 60% right, refinement is the critical missing piece. This decision should be made after Weekend 3-4 based on real fill quality.

**Depends on / blocked by:** v1 fill loop working reliably (Weekend 3 + 4 validation).
